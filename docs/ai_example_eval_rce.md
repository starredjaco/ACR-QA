## Vulnerable Code Snippet

```javascript
// core/appHandler.js — ping feature (connectivity test)

module.exports.ping = function (req, res) {
     exec('ping -c 2 ' + req.body.address, function (err, stdout, stderr) {  // ← LINE 39
             output = stdout + stderr
             res.render('app/ping', {
                     output: output
             })
     })
}
```

**OWASP Category:** A1 — Injection (Command Injection)
**Exploit:** An attacker sends `address = "x; id"` → the shell executes `ping -c 2 x; id` →
arbitrary command execution on the server.

## AI Explanation (Live — Cerebras API, 683ms)

**Explanation:**

This code violates **SECURITY-021 Shell Injection Risk** because it passes user input directly
into a command without proper sanitization, enabling command injection.

**WHY:** This matters because `shell=True` passes the command through the shell, enabling
command injection if any part of the command comes from user input.

## AI Autofix Suggestion

```javascript
// BEFORE (vulnerable):
exec('ping -c 2 ' + req.body.address, function (err, stdout, stderr) {
    output = stdout + stderr
    res.render('app/ping', { output: output })
})

// AFTER (fixed — use execFile with argument array, no shell interpolation):
exec(['ping', '-c', '2', req.body.address], function (err, stdout, stderr) {
    output = stdout + stderr
    res.render('app/ping', { output: output })
})
```

**Why this fix works:**
Passing an array to `execFile` (or `exec` with argument array syntax) prevents shell
interpolation. The OS treats each element as a separate literal argument — a semicolon in
`req.body.address` can never escape into a new command.
